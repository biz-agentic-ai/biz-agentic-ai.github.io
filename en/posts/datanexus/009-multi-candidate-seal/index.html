9. Nine Experiments and What Got Ruled Out: More Candidates Wasn't the Answer

April 19, 2026 · 13 min · Junho Lee | Translations:
  • Ko
Table of Contents
  • The one-line conclusion
  • First attempt: give the LLM more hints (4 failures)
  • Second attempt: generate multiple candidates and pick one (3 failures)
  • Third attempt: force diversity at the system level (3 failures)
  • Closed directions, remaining directions
  • What the 9 failures left behind
  • What’s next

In the previous post I wrote about pushing NL2SQL accuracy to 80% on a 30-question benchmark. I thought that was the milestone. Then I ran against BIRD Mini-Dev and got different numbers.

BIRD (Big Bench for Large-Scale Database-Grounded Text-to-SQL Evaluation) is an NL2SQL benchmark published in 2023 by researchers from Hong Kong Polytechnic University and Alibaba. Compared to older benchmarks like Spider, it uses much larger and messier databases — tens of thousands to hundreds of thousands of rows, with missing values, typos, and inconsistent formatting mixed in. It’s the standard comparison metric that NL2SQL papers cite.

BIRD splits into 9,428 training questions and 1,534 dev questions. Running the full dev set every iteration is expensive and slow. So Google and Alibaba jointly curated a Mini-Dev version: 50 representative questions each for MySQL, PostgreSQL, and SQLite. It’s designed for fast iteration. I used the SQLite 50.

The 50 questions span three databases: california_schools (12 questions), financial (11), and superhero (27). The superhero DB holds attributes of comic book characters — Marvel, DC, and others — including eye color, hair color, skin color, gender, and alignment. All questions are in English. None of it overlaps with the retail domain I trained on.

The result was 56%. To be precise: that 56% already includes DataHub Glossary and Vanna RAG Store injection. Zero-shot, with nothing added, was ~44%. Adding metadata got me +12 percentage points. Multi-candidate was an attempt to push past the 56% ceiling from there.

So I tried the multi-candidate approach: instead of asking the LLM to generate SQL once, generate k candidates and pick the best one. This has been the dominant direction in NL2SQL research for the past few years. Two core assumptions underlie it:

  1. There’s an accuracy ceiling on single-shot LLM output.
  2. That ceiling can be broken by splitting the problem into exploration + selection — raise the temperature for diversity, compare execution results, and pick the best.

The main implementations are self-consistency (sample k times from the same prompt and take a majority vote), execution-based selection (run candidates against the DB and compare results), and multi-agent pipelines like MAC-SQL or CHESS. All of them are variations of “generate multiple candidates, then pick one somehow.” The logic — widen the search space, filter with a verification signal — was compelling.

External data pointed the same direction. Four of the top five systems on the BIRD leaderboard use multi-candidate or multi-agent approaches. The top system from Ant Group scores 81.67%, second place is 80.88%. With SOTA tilted that way, there was no obvious reason not to try.

Internal review, literature survey, and cross-validation with AI advisors all pointed the same direction. The result: 9 experiments, 9 failures. This post is the postmortem.

The one-line conclusion#

This wasn’t a problem of generating more candidates. It was a problem of generating the right answer on the first try.

Comparing only execution results can’t tell you which SQL is more correct. Choosing the right columns and tables — schema binding — has to happen before execution. The direction shifted from “pick the best answer after execution” to “get the schema understanding right in the first place.”

First attempt: give the LLM more hints (4 failures)#

The initial idea was simple. Give the LLM more context when passing it a question, and accuracy will improve. I added four things in sequence: aggregation function hints (which of AVG/SUM/COUNT to use), term-to-column mapping hints (e.g., “revenue maps to this column”), column filters matching the aggregation, and a Top-K column candidate list.

Accuracy held at 83% on my own 30 questions, but four straight experiments showed no improvement on BIRD.

The first experiment had a bug on top of everything else — a code error meant the hints weren’t being passed to the LLM at all. After fixing it, the hints actually got through in the second experiment, and the self-test score dropped from 83% to 73%. The hint said “use average price,” so the LLM switched from the product table’s price column (products.price) — which it had been choosing correctly — to the order items table’s unit price column (order_items.unit_price). The LLM was already making the right choice; the hint overrode it. The same pattern appeared with a hint about growth rate calculations: the hint changed the SQL structure, and the new structure didn’t match the expected answer.

I reworked the hint logic to recover the lost accuracy. The self-test score came back, but BIRD stayed flat. The reason was straightforward: my term-to-column mappings were built for a retail domain. BIRD’s school, financial, and superhero databases don’t have them. The hints can only help when they exist.

The last attempt added a list of “candidate columns relevant to this question.” Looking at 19 failing questions, 60% of the failures weren’t about choosing the wrong column. They were about which SQL pattern to write. For example, “what fraction of all X matches condition Y” can be written as SUM(CASE WHEN condition THEN 1 ELSE 0 END) * 100 / COUNT(*) or as COUNT(CASE WHEN condition THEN 1 END) * 100 / COUNT(*). Both are valid SQL, but NULL handling can produce different results, and if the form differs from the expected answer, the grader marks it wrong. No amount of accurate column candidates can change which SQL pattern the LLM prefers.

That led to a fifth idea: inject the correct SQL pattern directly. Before implementing anything, I looked at 17 actual failing samples first. The result was surprising. In 7 out of 8 cases, the LLM was already writing the correct pattern (SUM CASE WHEN). There wasn’t a single case where it was using a wrong pattern. The problem I was trying to fix didn’t exist. This direction was dropped without writing a line of code.

“Four experiments in a row with no BIRD improvement. The fifth idea was rejected by the data before implementation. If nothing’s changed, trying again won’t change the result.”

That closed off the “give the LLM hints or context and let it decide” direction entirely.

Second attempt: generate multiple candidates and pick one (3 failures)#

The second idea was more direct. Generate k SQL candidates, run them all against the DB, compare results, pick the most plausible one. I started with tight constraints: k=3, temperature 0.3, no prompt changes, five heuristics.

Accuracy: 56%. Exactly the same as before. The breakdown explained why immediately.

MetricValue
Candidates converging to the same result46/50 (92%)
First candidate selected every time50/50 (100%)

The selector never overrode the first candidate. Not once. API calls tripled, accuracy stayed flat. At temperature 0.3, k=3 produces nearly identical outputs.

One other thing surfaced in this experiment. The MVP appeared to have improved from 48% to 56% — a +8 percentage-point gain. But when I re-graded the old baseline with the current grader, it turned out the old result was already 56%. The grader logic had improved slightly over time; it wasn’t the multi-candidate approach that moved the number. The +8pp was an illusion. That discovery got a “grader drift guard” baked in as permanent infrastructure. Without it, a failed experiment would have been recorded as a success.

I raised the temperature to 0.8 and added five prompt variations (v2). Surface diversity appeared — the rate of all candidates producing identical SQL dropped to zero. But result diversity stayed low: 62% of questions had all five candidates converging to the same answer. The original prompt (v0) won 47/50 (94%) of the time; the four variations combined for just three wins.

The reason:

There is one correct answer. The LLM always finds it. Changing the surface — aliases, JOIN order — just produces different paths to the same result. The SQL looks different; the execution result is the same.

For the third run (v3), I added a MANDATORY directive: “use this column, not that one.” A three-question spot check took 16 seconds and returned the same result all three times.

One question was telling. The LLM followed the directive and used the forced column — but the two columns returned the same values anyway. It followed instructions; the answer didn’t change. Forced directives couldn’t change the answer either.

That triggered the stop criterion. “Let the LLM freely generate multiple candidates” was closed.

Third attempt: force diversity at the system level (3 failures)#

LLM discretion wasn’t producing meaningful diversity. What if the system forced different columns and tables and generated multiple candidates that way?

A small-scale validation first. Forcing correct columns: 3/5 (60%) passed. Forcing wrong columns: 0/6 (0%) passed. A 60-percentage-point gap. The LLM followed forced directives 11 out of 11 times.

One question was the clearest. “What fraction of female superheroes have blue skin?” The correct answer requires joining on skin_colour_id. Joining on eye_colour_id (eye color) or hair_colour_id (hair color) produces wrong results. Picking the right column — schema binding — determines accuracy. The numbers confirmed it.

So far so good. Next was selector validation: can the selection logic actually identify the correct SQL from among the candidates? For each of 15 questions, I created four SQL candidates: correct, obviously wrong, plausibly wrong, and subtly different. The question was whether the selector could pick the right one. It stopped at question 7. Selection accuracy: 28.6%. Effectively random. The pre-committed stop criterion was 30%.

The reason was unexpected. The consensus rule — “if multiple candidates agree on a result, it’s correct” (R4) — was firing in the wrong direction. R4 triggered in 5 of 7 cases, and 4 of those 5 selected the wrong answer. A typical case: the financial DB stores transaction types in Czech. The correct SQL needs VYBER (Czech for “cash withdrawal”). But all four candidates used guessed English or transliterated values like VYDAJ KARTA or cash withdrawal. None of those values exist in the DB, so all four returned empty results. “Four candidates agree on a result” — the rule fired, and four wrong answers voted each other into first place. The consensus assumption requires “try enough times and the right answer will appear.” Here, the right answer never appeared. More agreement just confirmed the wrong answer.

The question was which of two things was broken: the selector logic, or the absence of any detectable signal in the first place. I built both at once to find out.

One was a V2 selector: scoring candidates by result row count (how many rows came back), aggregate function usage, empty result handling, and a diversity penalty.

The other was a signal classifier that categorized each question by whether a detection signal existed at all — from STRONG (signal clearly points to the right answer) to MISLEADING (signal actively points to the wrong one). The goal was to separate “V2 failed because of bad selector logic” from “V2 failed because there was no signal to detect.”

Results after running all 15:

MetricValue
STRONG signal rate8/15 (53.3%)
V1 overall accuracy40.0%
V2 overall accuracy33.3%
V2 − V1-6.7pp
V2 accuracy on STRONG cases37.5%
V2 theoretical maximum26.7%

V2 performed worse than V1. Making it more sophisticated made the score drop. Even with 53% STRONG signal — a reasonably good environment — V2 only hit 37.5%. The theoretical maximum of 26.7% comes from “STRONG 53% × 37.5% + random on the rest.” Beating the existing 56% was structurally impossible.

One question illustrated the core problem most clearly. In a STRONG signal case, V1 selected the correct answer; V2 selected the obvious wrong answer. V2’s scoring: correct SQL 55 points, the other three 75 points each. V2 ranked the correct answer lowest. In another question, the logic that “more result rows means wrong answer” gave -20 points to the correct SQL (74 rows) and nothing to the wrong one (1 row).

V2 kept repeating the same pattern: high scores for wrong SQL, low scores for correct SQL. Adjusting weights produces the same problem in different cases. Running the SQL and looking only at the output can’t tell you which candidate is more correct. The “pick the answer after execution” approach was closed here.

I committed four lines to the decisions log: “No combination of k>1 candidate generation + any selection method, ever. No selector weight-tuning experiments. No ‘just a bit more tuning’ micro-optimization. No retroactive threshold adjustment to reinterpret results as passing.” If someone brings a similar proposal in the future, those four lines are the immediate response.

Closed directions, remaining directions#

All three directions are closed.

Hint injection (4 failures): Any form of giving the LLM hints, candidate lists, or context and letting it decide.

LLM-driven multi-candidate (3 failures): Letting the LLM freely generate k≥3 candidates and picking from them — whether soft variation or forced directives.

Execution-based selection (2 failures): Any approach that looks only at execution results to retroactively pick the correct answer, whether rule-based or score-based.

There are exceptions. If the model changes (reasoning-specialized models, etc.), a learned selector gets introduced, or pre-execution signals (reasoning traces, schema confidence) are used, those are separate hypotheses. What’s closed here is the specific form of “post-execution selection based only on execution results.”

Four directions remain open: Schema grounding (getting it right on the first try), reasoning quality improvement, selective application (STRONG signal cases only), and model upgrade. They’re open because they haven’t been tested, not because they’re proven to work.

What the 9 failures left behind#

The experiments all failed. The infrastructure didn’t.

harness_corrections.py. Grader drift corrections with 13 tests attached. Without this, the MVP would have been recorded as a +8pp success.

signal_classifier.py. Four-level signal classification. Separates “weak selector” from “no signal.” This direction is closed, but the code stays for reference.

forced_branching.py + verify_compliance. Automatically checks whether the SQL the LLM generated actually uses the column it was told to use (using the sqlglot SQL parser). The early validation result — “11 out of 11 directives followed” — came from this code. Reusable as-is for the next schema grounding experiments.

Methodology was also preserved: hard-coding thresholds before running benchmarks to prevent post-hoc rationalization, using small spot experiments to make large decisions (v3 closed in 16 seconds), pre-committing stop criteria, A/B simultaneous comparison (V1 vs V2), grader drift guards. These became the default experimental design going forward.

9 experiment result JSONs and 5 experiment reports remain. When someone suggests trying multi-candidate again, the numbers are right there.

After the experiments concluded, I ran separate deep research sessions with Claude, ChatGPT, and Gemini. Of the 10 suggestions I received, 8 directly conflicted with approaches already closed: multi-candidate, ensemble, trained selectors. SOTA points that way, so the recommendation makes sense. Without the data from 9 experiments, I would have followed those suggestions.

What’s next#

Next direction: MVP features + schema understanding redesign

  • User-facing features first: Chat UI, DataHub integration
  • Improve accuracy in selecting the right columns and tables
  • Only single-shot paths are valid

No more multi-candidate refinement. The direction is getting it right on the first try.

The early third-attempt validation was decisive. When the correct binding is forced, the LLM follows it 100% of the time. The LLM is already good at “write SQL given a schema.” The problem is one step earlier: “correctly identify which columns and tables to use.” Every attempt to correct that step after the fact, through candidate selection, failed nine times. The conclusion is to improve schema understanding directly.

The core hypothesis for next time is a Schema Binding Plan. Instead of generating SQL directly, ask the LLM to first output a JSON specifying the tables, columns, and join conditions it plans to use — then have the system validate that plan before generating SQL. This isn’t “give hints and let the LLM decide.” It’s the system enforcing binding directly. The observation from the third attempt — “forced binding is followed 100% of the time” — is what this is built on.

DataHub integration, an embedding-based column ranker, and DDL grounding go on top of that. The 90% target sits on all of it.

One more thing to measure first. BIRD has only been tested with gemini-flash-lite as the model. On my 30-question self-test, gemini-2.5-pro beat flash-lite by +3.33pp — but it’s never been measured on BIRD. Checking the model tier before adding more techniques is the right order.

We’ll see.

In post 8 I ended with “80% is the start.” Looking back from post 9, the 80% was domain-specific. On an unseen domain: 56%. That’s the real starting point.
