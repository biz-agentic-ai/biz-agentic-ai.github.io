8. NL2SQL 정확도 66%에서 80%까지, 4번의 측정-개선 루프

4월 14, 2026 · 6 분 · Junho Lee | 번역:

목차

  • “상위 10명” 함정: Fake Determinism
  • 사이클 2: top-N 패치 + sanity check

이전 글 에서 “질문이 들어오면 그래프를 탈지, SQL을 짤지, 벡터 검색을 돌릴지를 누가 정하나"라는 라우팅 설계를 다뤘다. 이번 글은 그 설계를 실제로 구현하고, 30문항짜리 벤치마크를 돌리면서 EX(Execution Accuracy)를 66.67%에서 80%까지 올린 과정이다.

방법 자체는 단순했다. 30문항을 한 번 돌리고, 틀린 쿼리를 유형별로 나눈 다음 영향이 큰 부분만 손봤다. 그리고 다시 돌렸다. 이 과정을 네 번 반복했다.

시작점: 모든 쿼리가 LLM으로 간다#

Phase 0.5에서 Vanna에 DDL과 Glossary YAML(25개 비즈니스 용어)을 학습시킨 상태였다. PoC DB는 전자상거래 샘플로 테이블 21개, 행이 23만 개쯤 된다. 여기에 맞춰 30문항 테스트셋을 만들고 EX를 처음 측정했더니 66.67%가 나왔다. 30개 중 20개다.

막상 돌려보니 한 가지 문제가 바로 드러났다. 쉬운 쿼리든 어려운 쿼리든 전부 LLM이 SQL을 생성했다. “이번 달 총 매출은?” 같은 단순 집계도, “분기별 YoY 성장률을 구해줘” 같은 복잡한 쿼리도 둘 다 Vanna가 처음부터 끝까지 만들었다. LLM이 만든 SQL은 같은 질문에도 결과가 조금씩 달라질 수 있다. 단순 집계처럼 패턴이 뻔한 쿼리까지 LLM을 태울 이유가 없다.

그래서 QueryRouter를 만들었다.

사이클 1: QueryRouter 도입#

QueryRouter는 쿼리를 세 가지로 분류한다.

  • DETERMINISTIC: 패턴이 정해진 쿼리. “총 매출 합계”, “카테고리별 주문 건수"처럼 SQL 템플릿으로 풀 수 있는 것들. LLM을 안 거치고 바로 SQL을 만든다
  • HYBRID: glossary 용어가 포함되어 있으면서 조건이 복잡한 쿼리. 규칙으로 SQL 초안을 만들고 LLM이 검증한다
  • PROBABILISTIC: 위 두 가지에 해당하지 않는 자유 형식 쿼리. Vanna에 그대로 위임한다

MVP라서 분류기를 거창하게 만들지 않았다. 키워드 매칭과 정규식으로 분기한다. “합계”, “총”, “평균” 같은 키워드가 있으면 DETERMINISTIC 후보로 보고, glossary에 등록된 용어가 쿼리에 포함되어 있는지 체크한다.

30문항을 돌렸더니 EX 70%. +3.33%p. 숫자만 보면 미미한데, 안을 들여다보면 의미 있는 변화가 두 개 있었다.

하나는 동의어 인식률이 33%에서 67%로 뛴 것이다. “매출"을 “Revenue"로 물어봤을 때 이전에는 못 알아듣던 걸 Router가 glossary의 동의어 목록을 보고 잡아낸다.

다른 하나는 P95 응답시간이 26초에서 3.3초로 87% 줄어든 것이다. DETERMINISTIC 경로는 LLM을 안 거치니까 SQL 템플릿을 바로 실행한다. 비용도 줄고 속도도 빠르다.

근데 여기서 예상 못한 문제가 나왔다.

“상위 10명” 함정: Fake Determinism#

DETERMINISTIC으로 분기된 쿼리가 20개였는데, 그중 7개가 틀렸다. 35%다.

대표적인 게 이런 케이스다. “쿠폰 사용 상위 10명 고객"이라는 질문이 들어왔다. Router가 “상위"라는 키워드를 보고 HIERARCHY_ANCESTORS 템플릿으로 보냈다. 계층 구조에서 상위 노드를 탐색하는 템플릿이다. 근데 실제 의도는 ORDER BY coupon_count DESC LIMIT 10이다. 랭킹 쿼리지, 계층 탐색이 아니다.

“상위"라는 단어가 두 가지 의미를 갖는 거다. “상위 카테고리"라고 하면 계층 구조의 위쪽 노드를 말하고, “상위 10명"이라고 하면 랭킹 상위를 말한다. 키워드 매칭으로는 이걸 구분할 수 없다.

더 안 좋은 점은, 이게 LLM fallback보다 더 나쁘다는 거다. LLM은 맥락을 보고 “아, 이건 랭킹이구나” 하고 올바른 SQL을 생성할 가능성이 있다. 근데 DETERMINISTIC으로 분기되면 템플릿이 바로 실행되니까, 틀린 답이 검증 없이 나간다. 결정론적으로 틀리는 거다. ChatGPT에게 이 결과를 공유했더니 “Fake Determinism"이라고 이름을 붙여줬다.

사이클 2: top-N 패치 + sanity check#

이 시점에서 세 가지를 같이 수정했다.

우선 top-N 패턴을 정규식으로 잡아서 랭킹 표현이 먼저 매칭되면 HIERARCHY로 보내지 않도록 했다. 이 경우 HYBRID나 PROBABILISTIC으로 넘긴다.

TOP_N_PATTERNS = [
    r"상위\s*\d+",       # "상위 10명"
    r"top\s*\d+",        # "top 5"
    r"가장\s*(많|높|큰)", # "가장 많은"
]

이 규칙을 추가하니 Fake Det Rate가 35%에서 20%까지 내려갔다.

다음은 sanity check다. DETERMINISTIC 경로로 SQL을 실행한 뒤 결과가 비정상이면 LLM fallback으로 한 번만 재시도한다. “비정상"의 기준은 두 개, 결과가 0건이거나 NULL 비율이 70%를 넘는 경우다.

if det_result.row_count == 0 or det_result.null_ratio > 0.7:
    return await self._execute_probabilistic(query)  # 1회만 재시도

무한 재귀에 빠지면 안 되니까 fallback은 딱 한 번만 한다. PROBABILISTIC 결과가 나오면 그게 맞든 틀리든 그대로 반환한다.

마지막으로 few-shot exemplar 보강. 틀린 쿼리들의 오류 유형을 분류해보니 wrong_mapping(테이블/컬럼 매핑 오류)이 6건, wrong_formula(계산식 오류)가 2건이었다. 각 유형에 맞는 SQL 예시를 Vanna 학습 데이터에 추가했다. “이번 달 매출은?“이라는 질문에 DATE_TRUNC 패턴을 쓰는 SQL 예시, “순매출"을 구할 때 할인과 반품을 빼는 공식 예시 같은 것들이다.

결과는 EX 76.67%. +6.67%p. HYBRID 분기가 1건에서 6건으로 늘었다. Router가 “확신 없으면 LLM에 넘긴다"는 방향으로 움직인 건데, 설계 의도대로다.

사이클 3: is_active 함정, 그리고 정답이 틀릴 수 있다는 것#

wrong_mapping 중에 한 패턴이 계속 잡혔다. 고객 테이블을 조회할 때 is_active=true 필터를 안 넣는 거다. 비활성 고객(탈퇴, 휴면)이 포함되면 집계가 달라진다. few-shot으로 예시를 여러 번 줘도 안 먹혔다.

이건 “지식"이 아니라 “정책"이다. “활성 고객만 집계한다"는 건 비즈니스 규칙이지, LLM이 맥락에서 유추할 성격의 정보가 아니다. 그래서 sqlparse로 SQL을 파싱해서, is_active 컬럼이 있는 테이블이 쿼리에 포함되면 WHERE 조건을 자동으로 삽입하는 hard rule을 만들었다.

단순 문자열 치환이 아니라 SQL AST 레벨에서 처리했다. subquery 안에는 건드리지 않고, GROUP BY 뒤에 조건이 붙는 실수도 안 생긴다. DET/HYBRID/PROB 모든 경로에서 SQL 생성 직후, 실행 직전에 적용된다.

is_active rule을 넣고 벤치마크를 돌렸다. EX가 56.67%로 떨어졌다.

처음에는 결과를 잘못 본 줄 알았다. 8건이 PASS에서 FAIL로 돌아간 건데, 전부 같은 원인이었다. gold SQL(정답 SQL)에 is_active=true 조건이 없었다. 시스템은 비즈니스 정책에 맞게 필터를 넣었는데, 정답지가 그 정책 없이 작성되어 있으니 strict 비교에서 불일치로 판정된 거다.

여기서 드러난 건, 평가 기준 자체가 틀릴 수 있다는 거다. BIRD 벤치마크에서도 비슷한 사례가 있는데, strict scoring이 인간 전문가 판단과 62%만 일치하고 나머지 38%는 대부분 맞는 SQL을 틀렸다고 판정하는 false negative였다.

사이클 3.1: gold SQL 수정 → 80%#

작업은 크게 어렵지 않았다. gold SQL 8건에 is_active=true 조건을 추가했다. 시스템이 생성하는 SQL과 정답 SQL의 정책이 일치하도록 맞춘 거다.

재측정 결과 EX 80.00%. 목표로 잡았던 80%에 도달했다.

Phase 0.5부터 여기까지의 흐름을 표로 정리하면 이렇다.

버전EX핵심 변화틀린 건수
Phase 0.566.67%Vanna + Glossary RAG10/30
v170.00%QueryRouter 추가9/30
v276.67%top-N 수정 + sanity check + few-shot7/30
v3.180.00%is_active hard rule + gold 정합화6/30

+13.33%p 누적 개선. easy 난이도는 100%, Fake Det Rate는 35%에서 13.3%까지 내려갔다. hard 난이도는 37.5%로 변화 없는데, formula/JOIN 복잡도 문제라 DataHub synonym 확장이나 exemplar 보강이 필요한 영역이다.

이 과정에서 배운 것들#

is_active 필터 케이스에서 분명해졌다. 이건 LLM에 맡길 문제가 아니다. few-shot으로 가르치려고 여러 번 시도했는데 안 먹혔다. 확률적으로 처리할 문제가 아니다. sqlparse로 만든 hard rule 하나가 few-shot을 여러 번 추가하는 것보다 효과가 확실했다. “고객 테이블 조회 시 is_active=true 필터 필수"는 지식이 아니라 정책이고, 정책은 시스템에서 강제하는 게 맞다.

이번 케이스에서는 평가 기준이 오히려 문제였다. gold SQL이 비즈니스 정책을 반영하지 않으면, 올바른 개선이 오히려 점수를 떨어뜨린다. 테스트셋을 처음 만들 때 “우리 시스템의 정책이 gold SQL에 반영되어 있는가"를 체크하는 절차가 필요했다. 이번에는 8건밖에 안 됐지만, 테스트셋이 수백 개가 되면 이런 불일치를 나중에 잡기가 훨씬 어려워진다.

변수 하나씩 고정해서 보지 않았으면 원인을 분리하기 어려웠을 것이다. Router를 넣을 때, is_active를 넣을 때, 매번 같은 30문항으로 돌려서 delta를 봤다. 이렇게 하지 않았으면 “어디서 올랐는지, 어디서 깨졌는지” 구분이 안 됐을 거다. 특히 is_active 문제는 변수 분리 측정을 안 했으면 “EX가 떨어졌으니 is_active rule이 잘못됐다"고 판단했을 수 있다. 실제로는 rule은 제대로 동작했고 gold SQL이 문제였는데. baseline을 고정하고 한 번에 하나만 바꾸는 게 지루해 보여도 나중에 분석이 가능해진다.

Fake Determinism은 규칙 분류기의 구조적 한계다. “상위"가 계층인지 랭킹인지를 키워드 매칭으로는 풀 수 없다. top-N 패턴 제외 같은 예외 규칙으로 대응하긴 했지만, 이런 예외가 계속 늘어나면 규칙이 복잡해져서 유지보수가 어려워진다. Phase 2에서 LLM 분류기를 검토해야 할 수도 있다. 당장은 보류해 둔 상태다.

오류를 유형별로 나누고 나서야 어디를 고쳐야 할지 보였다. 틀린 쿼리를 그냥 “틀림"으로 놔두면 뭘 고쳐야 할지 모른다. wrong_mapping, wrong_formula, wrong_aggregation, hallucination 같은 유형으로 분류하니까 “wrong_mapping이 6건이니 매핑 exemplar를 보강하자”, “wrong_formula가 2건이니 계산식을 glossary에 명시하자” 식으로 다음 액션이 명확해졌다. 분류 없이 “정확도를 올려야 한다"만 가지고는 어디를 건드려야 할지 감이 안 온다.
