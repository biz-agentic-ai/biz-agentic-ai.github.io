## 익명화 규칙

- 구체적 회사명 금지. "대형 유통사", "대기업 제조사", "식품 계열사", "이전 직장" 등으로 대체
- 프로젝트 금액/규모 금지. "대규모 프로젝트", "수백 명 투입" 등으로 대체
- 데이터 사이즈(TB, 건수 등) 금지
- 구체적 컴포넌트명 금지. "메타데이터 카탈로그", "NL2SQL 엔진", "문서 지식엔진", "그래프 DB" 등 역할 기반으로 표현
- 단, DataNexus 자체 이름과 일반 기술 용어(NL2SQL, GraphRAG, Ontology, SKOS, LLM 등)는 사용 가능

## 톤 규칙

- 문체: "~다", "~했다", "~이다" (블로그 서술체, 존댓말 아님)
- 전문적이되 딱딱하지 않게. 동료 엔지니어에게 설명하는 느낌.
- 감성적 표현 금지. 느낌표 금지. 이모지 금지.
- 자조적 유머는 허용하되 과하지 않게.
  예) "결국 6개월 동안 메타데이터 정리만 하다가 팀원들한테 엄청 욕먹었다."

## 문장 패턴 규칙 (AI 티 제거)

### 금지 패턴
- "결론적으로", "종합하면", "요약하자면"
- "이유는 명확하다", "이유는 단순하다", "이유는 간단하다"
- "첫째, 둘째, 셋째" 나열
- "~에 대해 알아보겠습니다", "~를 살펴보겠습니다"
- "게임 체인저", "혁신적", "놀랍게도", "드디어"
- "또한", "그러나", "따라서"가 3회 이상 반복
- 매 문단 첫 문장이 같은 구조로 시작
- 5문장 이상 비슷한 길이가 연속

### 의도적으로 할 것
- 짧은 문장(5~10자)과 긴 문장(40~60자)을 섞기
- 가끔 한 줄짜리 문장으로 리듬 끊기
- 연결어 없이 문장을 이어붙이기 (맥락으로 연결)
- 구체적 수치/이름/날짜 넣기 (추상적 표현 대신)
- "~인데", "~는데" 같은 구어체 연결은 허용

## 구조 규칙

- 하나의 글에 하나의 주제
- 도입은 상황/장면으로 시작 (개념 정의로 시작하지 않기)
- 본문에 최소 1개의 구체적 경험 에피소드 포함
- 코드/다이어그램은 필요한 만큼만, 장식용 금지
- 마무리는 "다음 글에서 ~를 다룬다" 한 줄이면 충분

## 다이어그램 스타일 규칙 (Excalidraw)

블로그에 삽입하는 모든 다이어그램은 Excalidraw로 작성하며, 아래 속성을 일괄 적용한다.

### 필수 속성
- `roughness: 0` — 손그림 효과 제거, 깔끔한 직선/곡선
- `fontFamily: 2` — Helvetica (sans-serif), 전문적 톤 유지
- `roundness: { "type": 3 }` — 모서리 라운드 처리 (모든 사각형)
- `fillStyle: "solid"` — 배경색 solid fill

### 색상 팔레트 (일관성 유지)
| 용도 | 배경색 | 테두리색 |
|------|--------|----------|
| 사용자 입력/데이터 소스 | `#a5d8ff` (연한 파랑) | `#4a9eed` |
| 처리/라우팅 | `#d0bfff` (연한 보라) | `#8b5cf6` |
| 온톨로지/지식 | `#c3fae8` (연한 틸) | `#22c55e` |
| NL2SQL/변환 | `#fff3bf` (연한 노랑) | `#f59e0b` |
| 결과/출력 | `#b2f2bb` (연한 초록) | `#22c55e` |
| 에러/경고 | `#ffc9c9` (연한 빨강) | `#ef4444` |
| 영역 배경 (zone) | opacity 25~35% 적용 | — |

### 폰트 크기
- 다이어그램 제목: `fontSize: 24`
- 박스 라벨: `fontSize: 16~18`
- 영역 라벨/보조 설명: `fontSize: 14~16`
- 보조 설명 텍스트 색상: `#757575` (회색, 본문과 구분)

### 화살표
- `strokeWidth: 2`, `roughness: 0`
- 점선 화살표(`strokeStyle: "dashed"`)는 동기화/간접 연결에만 사용
- 화살표 색상: 주 흐름은 `#1e1e1e`, 보조 흐름은 `#757575`

## 경험 원재료 (프롬프트에 삽입용)

아래는 실제 프로젝트에서 겪은 경험이다. 글에 자연스럽게 녹여 쓰되, 회사명/금액/규모는 익명화한다.

### DW/메타데이터 관련
- 대기업 제조사 DW 클라우드 마이그레이션 2건.
  테이블 수천 개, 컬럼명이 전부 약어. T_CUST_MST, T_ORD_DTL 같은 이름만
  보고는 비즈니스 의미를 파악할 수 없었다.
  같은 회사인데 사업부마다 테이블 네이밍 규칙이 달랐다.
  "매출"이 사업부마다 다른 테이블을 가리켰다.
- 식품 계열사 차세대 정보계. 다수 벤더, 1년 넘는 기간.
  멀티벤더 PMO를 하면서 각 벤더가 정의하는 "매출", "원가"의 기준이 달라서
  데이터 정합성 맞추는 데 상당한 시간을 썼다.

### NL2SQL 관련
- 대형 유통사 BI Agent. 자연어 질의/응답 시스템 구축.
  카드사/통신사/공공데이터까지 내외부 데이터 통합.
  NL2SQL 적용했는데, 벤치마크 정확도와 실환경 정확도의 갭을 직접 경험.
- 대화형 BI 솔루션 직접 개발. NL2SQL + GPT LLM + Gemini.
  프롬프트 엔지니어링 최적화, 멀티에이전트 아키텍처 설계.
  DDL만으로는 LLM이 비즈니스 맥락을 이해할 수 없다는 걸 확인.

### 조직/컨설팅 관련
- 이전 직장에서 10년 넘게 DW/BI 컨설팅. 팀을 직접 빌딩해서 키움.
  C레벨 대상 발표/세미나 수십 회.
  "왜 메타데이터가 중요한지" 설명하는 데 커리어의 상당 부분을 썼다.

## 사용 방법

1단계: 글의 주제/핵심 메시지를 한 줄로 정한다
2단계: 위 경험 원재료 중 관련 있는 것을 고른다
3단계: 이 문서 전체를 프롬프트에 포함하고 초안을 생성한다
4단계: 초안에서 AI 패턴을 손으로 깨뜨린다
5단계: 이준호 본인의 표현/에피소드를 추가한다
